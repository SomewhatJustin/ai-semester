I am also not surprised that Ollama was slower in getting responses. I'm working on an NVIDIA RTX 4060 with a strong wired ethernet connection.

One of my prompts is

> If all bloops are bloopsies and some bloopsies are glips, are some bloops necessarily glips? Answer yes/no with 1 sentence.

The answer should be "No". GPT-5 and gpt-oss:20b both got it right every time. From the last run of GPT-5:

> No; the bloopsies that are glips need not include any bloops.

Gemma3:4b, on the other hand:

> Yes, some bloops are necessarily glips because if all bloops are bloopsies and some bloopsies are glips, then the bloops that are also bloopsies must be glips.

The reasoning sounds ~right but falls down upon further inspection.

You can see more in the [results csv](../data/results/benchmark_results_20250903_151939.csv)

On the other extreme, at temperature 0, the model will only output what is the most likely character - essentially making it deterministic. Nothing interesting there. The responses barely differed between temperature 0 and 0.8 (the default).

The creative writing responses do seem to get better as you increase temperature, with the exception of temperature 5 where the output doesn't follow English conventions very well and adds many unnecessary newlines.

Top_p, on the other hand, is a way to eliminate options for the next token. This is a potential compute-saving measure. In top_p, it only considers tokens that fall into the top X% of likelihood to be the next character.

Sure enough, when we restrict top_p to 0.1 (a very small amount!), time to first token averages 0.17 seconds and total compute time is 3.97 seconds. Fairly quick compare to using a top_p of 0.995, where we see time to first token averaging 0.17 seconds and total time 4.99 seconds. Not a massive difference.

Top_k is fairly similar to top_p, but instead of using summing probability to determine which tokens should be considered candidates, the model just chooses X amount of tokens to be considered.

Top_p and top_k performed fairly similarly.
